# Обзор архитектуры

В структуре компании выделяются четыре подразделения:
- Головной офис.
- Клиники.
- ИИ-сервисы для работы с медицинскими данными в виде отдельной компании.
- Финтех-сервисы в виде отдельной компании с банковской лицензией.

Врамках кейса нужно поработать с такими к продуктами:
- Финтех-сервисы.
- Внутренняя медицинская система-DWH с клиентским интерфейсом и дополнительной бизнес-логикой под финтех и ИИ-сервисы.
- Много кастомизаций BI системы поверх DWH.
- ИИ-сервисы для работы с медицинскими данными.
- Слой интеграций через старую шину данных.

Сейчас развёрнуты такие технологии, сервисы и приложения:
- DWH на базе Microsoft SQL-сервера 2008 года.
- Power BI.
- Power Builder.
- Шина на базе Apache Camel.
- ИИ-сервисы на Python.
- Финтех сервисы на golang и Java.

# Проблемные места

1. **Устаревшая DWH (Высокий приоритет)**: Использование Microsoft SQL-сервера 2008 года может привести к проблемам с производительностью, масштабируемостью и безопасностью. Это может замедлить работу всей системы и увеличить риски утечки данных. Кроме того, поддержка устаревших версий ПО может быть прекращена, что приведет к отсутствию обновлений и исправлений уязвимостей.
2. **Старая шина данных (Средний приоритет)**: Использование устаревшей версии Apache Camel может быть неэффективным для современных требований к интеграции данных. Рекомендуется обновить версию Apache Camel до последней стабильной версии для улучшения производительности и поддержки новых протоколов и стандартов. Также необходимо оптимизировать SQL-запросы и использовать индексы для ускорения выполнения запросов к DWH.
3. **Много кастомизаций BI системы (Низкий приоритет)**: Это может привести к сложности в поддержке и обновлении системы. Частые изменения и настройки могут вызвать ошибки и увеличить время на их исправление. Кастомизации могут сделать систему менее гибкой и затруднить внедрение новых функций и обновлений.

# Целевая архитектура

Переход на архитектуру Data Lakehouse, которая сочетает в себе преимущества Data Lake и Data Warehouse, может значительно улучшить управление данными и аналитические возможности компании. Основные преимущества Data Lakehouse:

1. **Гибкость хранения данных**: Data Lakehouse позволяет хранить структурированные и неструктурированные данные в одном месте, что упрощает управление данными и снижает затраты на хранение.
2. **Высокая производительность**: Использование MS SQL для DWH обеспечивает высокую производительность при обработке и анализе данных, а S3 для Data Lake позволяет эффективно хранить большие объемы данных.
3. **Масштабируемость**: Data Lakehouse легко масштабируется, что позволяет обрабатывать растущие объемы данных и поддерживать высокую производительность.
4. **Унифицированный доступ к данным**: Data Lakehouse предоставляет единый интерфейс для доступа к данным, что упрощает их анализ и использование в различных бизнес-приложениях.
5. **Снижение затрат**: Хранение данных в S3 позволяет значительно снизить затраты на хранение по сравнению с традиционными DWH решениями.
6. **Поддержка современных аналитических инструментов**: Data Lakehouse поддерживает интеграцию с современными аналитическими инструментами, такими как Power BI, Tableau и другими, что позволяет быстро и эффективно создавать отчеты и панели мониторинга.

## Рекомендации

- **Apache Iceberg**: Использование Apache Iceberg для управления метаданными. Это позволит улучшить производительность и надежность системы, а также упростить управление большими объемами данных. Apache Iceberg поддерживает ACID-транзакции и обеспечивает высокую производительность при работе с большими наборами данных, а также различные форматы данных и интеграцию с различными инструментами аналитики.
- **DataHub**: Внедрение DataHub для упрощения взаимодействия пользователей с системой. DataHub предоставляет удобный интерфейс для поиска, управления и анализа данных, что повышает эффективность работы пользователей и улучшает качество принимаемых решений. DataHub позволяет централизованно управлять метаданными, обеспечивая прозрачность и контроль над данными.
